### 개요

밥토리라는 이름은 `밥 + 스토리`에서 출발했다. 사용자가 자신의 식단과 건강 기록을 딱딱한 폼 입력이 아니라 이야기처럼 남길 수 있으면 좋겠다는 의미를 담았다.

건강 기록 서비스에서 가장 어려운 부분은 기능 자체보다 꾸준히 기록하게 만드는 일이라고 생각했다. 식단은 음식명, 끼니, 수량, 영양 정보가 필요하고 운동은 종목, 강도, 시간, 날짜가 필요하다. 매일 입력해야 하는 항목이 많아질수록 사용자는 금방 기록을 포기하게 된다.

밥토리는 이 문제를 AI 채팅으로 풀어보려 한 서비스다. 사용자는 "오늘 아침으로 싸이버거 먹었어", "저녁에 검도했어", "이 사진 점심으로 먹었어"처럼 자연스럽게 입력하고, AI는 그 문장에서 식단, 운동, 체중 정보를 추출해 기록 후보를 제안한다. 사용자는 제안 내용을 확인하고 수정한 뒤 실제 기록으로 저장할 수 있다.

즉, 밥토리는 단순한 AI 챗봇이 아니라 `대화 -> AI 기록 제안 -> 사용자 확정 -> 식단/운동/체중 기록 저장 -> 캘린더/프로필에서 조회`로 이어지는 건강 기록 시스템이다.

### 개발 목표

- 사용자가 대화하듯 식단, 운동, 체중을 기록할 수 있도록 한다.
- AI가 추출한 정보를 바로 저장하지 않고, 사용자가 확인 후 확정하도록 한다.
- 식단, 운동, 체중, 일일 목표를 한 화면 흐름 안에서 관리할 수 있도록 한다.
- 음식 검색 실패와 음식 등록 요청을 관리자 검수 흐름으로 연결해 음식 데이터를 보강한다.
- Prometheus, Grafana를 통해 운영 상태를 확인할 수 있도록 한다.

### AI 챗봇 대화

AI 챗봇은 밥토리의 중심 기능이다. 사용자는 일반 채팅처럼 메시지를 입력하고, AI는 사용자의 건강 프로필, 일일 목표, 최근 기록, 채팅 요약, 사용자 메모리를 참고해 답변을 생성한다.

AI 채팅에서는 단순 답변뿐 아니라 식단, 운동, 체중 기록 후보를 추출한다. 예를 들어 사용자가 "나 오늘 아침으로 싸이버거 먹었어"라고 입력하면 AI는 아침 식단 후보를 만들고, 사용자는 제안 카드에서 음식 정보와 수량을 확인한 뒤 실제 식단 기록으로 저장할 수 있다.

이미지 업로드도 지원한다. 사용자가 음식 사진을 올리고 "점심으로 이거 먹었어"라고 입력하면 AI는 이미지와 텍스트를 함께 참고해 음식 정보를 추정하고 기록 후보를 만든다.

AI는 기록을 제안하고, 최종 저장은 사용자가 확정한다. 이 구조를 통해 AI 편의성과 데이터 정확도 사이의 균형을 맞췄다.

### 식단 및 운동 기록

사용자는 AI 채팅뿐 아니라 일일 기록 화면에서도 직접 식단과 운동을 기록할 수 있다.

식단 기록은 날짜, 끼니, 음식, 수량을 기준으로 저장한다. 동일한 날짜와 끼니의 기록은 갱신할 수 있고, 일일 기록 화면에서 하루 섭취 칼로리와 탄수화물, 단백질, 지방 합계를 확인할 수 있다.

운동 기록은 운동 종목, 강도, 운동 시간, 날짜, 메모를 기준으로 저장한다. 운동 종목에는 강도별 MET 정보가 연결되어 있어 사용자의 체중과 운동 시간에 따라 소모 칼로리를 계산할 수 있다.

체중 기록도 함께 관리한다. 날짜별 체중을 저장하고, 가장 최근 체중은 건강 프로필의 현재 체중과 동기화된다. 프로필 화면에서는 체중 추세를 확인할 수 있다.

### 음식 검색

음식 검색은 식단 기록의 정확도를 높이기 위한 기능이다. 사용자는 음식명으로 DB에 저장된 음식을 검색하고, 제공량과 영양 정보를 확인한 뒤 식단에 추가할 수 있다.

검색 결과가 없을 때는 검색 실패 기록을 남긴다. 이 검색 실패 데이터는 외부 음식 API에서 후보를 수집하는 기준이 된다. 또한 사용자는 직접 음식 등록 요청을 보낼 수 있고, 관리자는 이 요청을 검토해 공식 음식 데이터로 승인하거나 반려할 수 있다.

이 흐름을 통해 음식 데이터는 정적인 초기 데이터에 머물지 않고, 사용자의 검색 실패와 요청을 기반으로 점진적으로 보강된다.

### 일일 목표와 진행률

밥토리는 사용자의 키, 체중, 목표 체중, 목표 유형을 바탕으로 일일 섭취 목표와 운동 목표를 추천한다. 사용자는 추천값을 그대로 저장하거나 직접 수정할 수 있다.

채팅, 캘린더, 일일 기록, 프로필 화면에는 Today Panel이 표시된다. Today Panel은 오늘의 섭취 목표, 운동 목표, 섭취 영양소, 오늘 기록 요약을 보여준다. 사용자는 현재까지의 진행률을 보면서 오늘의 식단과 운동을 조정할 수 있다.

### 관리자 기능

관리자는 음식 데이터를 관리하는 역할을 한다.

사용자가 등록 요청한 음식은 관리자 페이지에서 조회할 수 있다. 관리자는 요청 내용을 검토하고, 영양 정보나 제공량을 보정한 뒤 승인할 수 있다. 부적절하거나 중복된 요청은 반려 사유와 함께 반려한다.

외부 음식 API에서 수집된 후보도 별도의 검수 대상이다. 관리자는 검색 실패 데이터 기반으로 수집된 외부 음식 후보를 확인하고, 적절한 후보를 공식 음식 데이터로 승인할 수 있다.

### 운영 및 모니터링

백엔드는 Actuator와 Micrometer Prometheus를 통해 운영 지표를 노출한다. Prometheus는 지표를 수집하고, Grafana는 이를 대시보드로 시각화한다.

모니터링 대상은 CPU 사용량, JVM heap, disk usage, request count, response time, failure ratio, AI request count, AI token usage 등이다. 이를 통해 단순히 기능 구현에서 끝나지 않고, 배포 이후 서비스 상태를 확인할 수 있는 운영 기반을 마련했다.

## 아키텍처

밥토리는 Vue 기반 프론트엔드와 Spring Boot 백엔드, PostgreSQL, Redis, 외부 AI API, OAuth 제공자, FatSecret API, Prometheus/Grafana로 구성된다.

![시스템 아키텍처](assets/readme/system-architecture.svg)

### 요청 흐름

1. 사용자는 Vue 프론트엔드에서 로그인, 채팅, 기록 관리 기능을 사용한다.
2. 프론트엔드는 Spring Boot 백엔드로 REST API 또는 SSE 요청을 보낸다.
3. 백엔드는 Spring Security와 JWT를 통해 인증/인가를 처리한다.
4. 주요 데이터는 PostgreSQL에 저장하고, Refresh Token과 Access Token blacklist는 Redis로 관리한다.
5. AI 채팅 요청은 AI API와 연동하고, 음식 후보 수집은 FatSecret API와 연동한다.
6. 운영 지표는 Actuator/Micrometer를 통해 Prometheus가 수집하고 Grafana에서 확인한다.

### 백엔드 구조

백엔드는 도메인별로 `controller`, `dto`, `entity`, `exception`, `mapper`, `service` 패키지를 나누었다. Controller는 HTTP 요청/응답과 인증 사용자 조회를 담당하고, Service는 비즈니스 흐름과 검증을 담당한다. Mapper와 MyBatis XML은 DB 접근만 담당하도록 분리했다.

사용자별 데이터는 클라이언트에서 전달한 userId를 신뢰하지 않고, 인증 컨텍스트에서 얻은 사용자 ID를 기준으로 조회, 수정, 삭제한다.

## 기술적 고민

### AI 응답을 바로 저장하지 않고 제안으로 분리

AI가 문장에서 식단이나 운동 정보를 추출하더라도, 그 결과가 항상 정확하다고 볼 수는 없다. 음식명, 수량, 날짜, 운동 강도는 사용자 의도와 다르게 해석될 수 있다.

그래서 AI 결과를 바로 DB에 저장하지 않고 제안 카드로 보여주도록 설계했다. 사용자는 제안된 정보를 확인하고, 필요한 경우 수정한 뒤 확정한다. 이 방식은 AI 기능의 편의성을 가져가면서도 최종 데이터 품질은 사용자가 통제하게 만든다.

### 음식 데이터 부족 문제

음식 검색 기능은 데이터가 충분하지 않으면 사용성이 크게 떨어진다. 프로젝트 초기에는 DB에 없는 음식이 많을 수밖에 없었다.

이를 해결하기 위해 검색 실패 기록을 저장하고, 반복적으로 검색되는 실패어를 스케줄러를 통해 외부 음식 API 후보 수집 대상으로 삼았다. 사용자가 직접 음식 등록 요청을 보내는 흐름도 추가했다. 관리자는 이 데이터들을 검수해 공식 음식 데이터로 반영한다.

### JWT와 Redis 기반 인증 흐름

Access Token은 짧게 사용하고, Refresh Token은 Redis에 저장해 재발급에 사용했다. 로그아웃 시에는 Refresh Token을 삭제하고 Access Token을 blacklist로 관리해 남은 유효 시간 동안 재사용을 막았다.

OAuth 로그인은 Google과 Naver를 지원하며, OAuth 신규 사용자는 서비스에 필요한 건강 프로필 정보를 추가로 입력하도록 온보딩 흐름을 구성했다.

### 운영 모니터링

단순히 배포만 하는 것이 아니라 서비스 상태를 확인할 수 있도록 Prometheus와 Grafana를 구성했다. 요청 수, 응답 시간, 실패율, JVM 메모리, AI 사용량 관련 지표를 확인할 수 있게 하여 운영 관점의 안정성을 확보하려 했다.

## 회고

### 잘한 점

음식 검색 실패의 경우에 자동으로 실패 검색어가 저장되고 음식 후보를 검색한 뒤, 관리자 승인을 통해 추가하는 흐름을 만들면서 데이터가 부족한 상황을 서비스 운영 흐름 안에서 해결하려고 했다. 이 부분은 단순 기능 구현보다 실제 서비스를 운영한다면 어떤 문제가 생길지를 고민한 결과였다.

하지만 급하게 추가한 기능이다보니 실제 운영환경에서 이 기능이 어떤 영향을 끼치게 될지에 대한 분석은 부족했다.

의미없는 실패 검색어에 대한 필터링이 제대로 되는지, 많은 사용자의 많은 실패 검색어가 들어올 때 실제로 이 기능이 제대로 작동하고 다른 기능에 영향을 끼치진 않을지에 대한 추후 분석이 필요하다고 생각한다.

### 어려웠던 점

내가 담당한 부분은 아니었지만, AI에게 사용자의 컨텍스트를 어떻게 전달하면 좋을지에 대한 부분이 가장 어려웠던 것 같다. 어떤 정보들을 어떻게 주어야 AI가 사용자에게 맞는 답변을 낼지에 대한것이 모호했다.

이 기능은 팀장인 채은 누나가 담당해서 직접 고민하며 구현하진 않았지만 채은 누나의 AI메모리 설계를 보며 배운점이 많았던 것 같다.

### 아쉬운 점

AI메모리가 잘 작동하는것은 확인했지만, 토큰 사용량을 줄이는법에 대해선 시간이 부족해 연구해볼 시간이 없었다. 현재는 API로 연결되어 모든 채팅에서 AI에게 사용자의 정보를 항상 전달하고 있어, 자동적으로 토큰 사용량이 많아질 수 밖에 없다.

또 모니터링을 추가하긴 했지만, 실제 많은 사용자가 들어오는 상황을 연출해보지 않았기 때문에 사실상 안쓰는 기능이나 마찬가지였다. 추후 부하 테스트 등을 통해 성능 최적화와 모니터링을 의미있게 진행해보고자 한다.

### 배운 점

이전까진 Spring의 근본적인 구조에 대해 모른채로 프로젝트를 진행해 그냥 되나보다~하면서 코드를 짰었다.

SSAFY에서 Spring의 구조에 대해 공부한 상태에서 진행한 프로젝트이다 보니 확실히 코드를 작성하며 이 코드가 어떻게 작동하는지 머릿속에서 전체구조가 잘 잡혔다. Spring AOP, Spring Security, scheduler등 배운것들을 많이 써볼 수 있어서 개발실력이 한층 늘어난것이 느껴지긴 했다.

하지만 배운만큼 모르는것들이 더 늘어나 여전히 모르는건 너무나도 많다. 마음이 급해지지만 급하다고 잘 배우는건 아니기 때문에 차근차근 하나씩 배워나가는 수밖에 없겠다. 다음은 DB쪽을 깊게 파볼까 생각중이다.
